### 2.4 年份定额文件格式（.de/.fz/.qt/.cl）

分布在 `Dek/2003/`, `Dek/2013/`, `Dek/2020/`, `Dek/2021/`, `Dek/2024/`, `Dek/2025/` 六个目录。

**文件头分析（xxd）**：均为二进制加密/私有格式，非标准数据库。推测：
- `.de` = 定额子项明细（最大 ~14.5MB）
- `.fz` = 辅助/分组数据（较小，~2KB-450KB）
- `.qt` = 其他/取费数据（中等，~30KB-5.4MB）
- `.cl` = 材料数据（最大，14.5MB）
- `.ft` = 部分定额集有，可能是附加费率

**格式特征**：无标准文件头（非SQLite/JSON/XML），数据经过编码，需逆向解析。

## 4. 现有 API 能力评估

### 4.1 Flask API (`api_server.py`) 端点清单

| 端点 | 方法 | 状态 | 说明 |
|------|------|------|------|
| `/health` | GET | ✅ 可用 | 健康检查，返回 dwgj.db 总记录数 |
| `/api/v1/定额集` | GET | ✅ 可用 | 列出所有定额集及记录数 |
| `/api/v1/定额集/<id>` | GET | ✅ 可用 | 某定额集下的定额列表 |
| `/api/v1/定额/search` | GET | ✅ 可用 | 关键词搜索定额（编号/名称/章节） |
| `/api/v1/定额/detail` | GET | ✅ 可用 | 精确查询定额 |
| `/api/v1/清单/search` | GET | ❌ 挂了 | QDK_DB 路径指向空的 `.qa` 文件 |
| `/api/v1/清单式/<id>` | GET | ❌ 挂了 | 同上，连接的数据库不对 |
| `/api/v1/配套` | GET | ⚠️ 逻辑有误 | 匹配算法过于粗暴（前2位匹配） |

### 4.2 核心问题

1. **清单数据源错误**：代码中 `QDK_DB` 指向 `Qdk_UpdateEPC.qa`（实际有数据），但该文件在代码中路径可能不对，或早期指向的是空文件。经确认 `Qdk_UpdateEPC.qa` 实际有 6.4MB 数据，修复路径后清单 API 应该可用。
2. **配套查询逻辑粗糙**：用清单编号前2位匹配定额章节，太宽泛，需改进。
3. **无分页支持**：定额集详情只有 limit，无 offset 分页。
4. **无写操作**：纯只读 API，不支持造价文件创建/编辑。

## 5. SaaS 重构关键发现与建议

### 5.1 关键发现

1. **数据结构清晰可迁移**：核心数据仅 2 个 SQLite 库（dwgj.db + Qdk_UpdateEPC.qa），schema 简单，总数据量 < 13MB，迁移成本低。
2. **年份定额文件是加密二进制**：.de/.fz/.qt/.cl 文件是私有格式，已有部分被解析入 dwgj.db，但完整解析需要逆向 EpointLib DLL 或找到官方数据导出接口。
3. **XML 配置丰富**：Config/ 下 29 个 XML 文件定义了完整的业务规则（地区代码、数据结构、UI配置等），是业务逻辑的宝贵参考。
4. **XSD 标准齐全**：23 个 XSD 文件覆盖了江苏省各市的招投标数据交换标准，可直接用于 SaaS 数据验证。
5. **报表模板成熟**：Report/ 下包含 20+ 报表类别（EPC报表、全费用报表、结算审核等），可参考设计 SaaS 报表。
6. **WPF 桌面应用**：59 个 DLL，主框架为 WPF + Caliburn.Micro MVVM，纯 Windows 应用。
7. **材料价信息**：`价目信息/` 目录有 2007/2015 年材料价（.tz 文件），CjzsCache.db 有缓存。

### 5.2 数据完整度评估

| 数据类别 | 完整度 | 说明 |
|----------|--------|------|
| 定额主数据 | ⭐⭐⭐⭐⭐ | 29 个定额集，12.3 万条，schema 清晰 |
| 清单项目 | ⭐⭐⭐⭐ | 2 万条清单 + 1.7 万条工作内容 |
| 定额组成/人材机 | ⭐⭐ | 存在于加密 .de/.cl 文件中，未入库 |
| 材料价格 | ⭐⭐ | 仅有部分年份缓存，无持续更新机制 |
| 取费标准 | ⭐⭐ | 存在于 .fz/.qt 文件中，未入库 |
| 报表模板 | ⭐⭐⭐⭐ | 完整，可直接参考 |

### 5.3 SaaS 重构建议

1. **分阶段迁移**：Phase 1 先做定额查询 + 清单查询（已有数据）；Phase 2 逆向解析 .de/.cl 获取组成数据；Phase 3 自建计价引擎。
2. **数据解耦**：将 29 个定额集拆分为独立数据包，按省份+年份管理，支持在线更新。
3. **API 优先**：REST/GraphQL API 设计优先于前端，让数据先行。
4. **参考 XSD 做数据标准化**：23 个 XSD 可直接生成数据模型。

## 6. 技术路线建议

### 6.1 数据库选型

| 方案 | 推荐度 | 理由 |
|------|--------|------|
| **PostgreSQL** | ⭐⭐⭐⭐⭐ | 成熟、JSON 支持、全文搜索、适合 SaaS 多租户 |
| MySQL | ⭐⭐⭐⭐ | 够用，但 PG 的 JSONB 和数组类型更适合定额数据 |
| MongoDB | ⭐⭐⭐ | 灵活，但定额数据本质是关系型的 |
| SQLite | ⭐⭐ | 仅适合单机/离线模式，不适合 SaaS |

**建议**：PostgreSQL + PostGraphile（自动生成 GraphQL API），或 PostgreSQL + Prisma + REST。

### 6.2 框架选型

| 层 | 方案 | 理由 |
|----|------|------|
| API | **FastAPI** (Python) | 类型安全、自动文档、性能好、与现有 Flask 代码兼容 |
| API 备选 | NestJS (Node.js) | 如果团队偏向 TypeScript |
| 前端 | **Next.js + shadcn/ui** | SSR + 组件化，适合复杂表单 |
| 计价引擎 | Python 库 | 纯逻辑，可独立部署为微服务 |
| 报表 | **Puppeteer/Playwright** | HTML→PDF，替代 Crystal Reports |

### 6.3 迁移路线图

```
Phase 1 (2-3周): 数据迁移 + 基础 API
  ├─ SQLite → PostgreSQL 脚本
  ├─ 定额查询 API (FastAPI)
  ├─ 清单查询 API
  └─ 简单前端页面

Phase 2 (4-6周): 逆向解析 + 组成数据
  ├─ 解析 .de/.cl 文件格式
  ├─ 人材机数据入库
  ├─ 取费标准入库
  └─ 综合单价计算引擎

Phase 3 (6-8周): 完整计价功能
  ├─ 造价文件 CRUD
  ├─ 报表生成
  ├─ XSD 数据导入导出
  └─ 多用户/多项目管理
```
